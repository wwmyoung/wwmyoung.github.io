    出发点

    自从国内Deepseek将AI大模型开源并且大幅降低使用费用之后，各大厂商也陆续推出了自家高性价比模型。借助AI迅起之势，各行各业都在探索智能化场景转型。根据网上现有的公开资料，面向高校学生的智能化场景，大多是AI教学，AI心理辅导，AI学情分析。对于个别公开的案例截图，能实际落地的并不多。

    从学生角度出发，我感觉现有方案很难落地有以下几种原因：

    • 现有场景基本集中在电脑端，但是手机移动端才是学生使用的主战场
    • 现有方案需要采集学生签到，打卡，心理做题测评等数据，这些都会引起学生反感
    • 学生性格各异，对于很多数据采集，都是胡乱填写或者隐藏自身性格
    • 基于上条，很多分析结果其实不准确，徒增辅导员工作量

    出于以上原因，面向学生的智能化场景，一方面要在很小侵入学生隐私情况下带来方便，另一方面减少辅导员的工作量。因此，实际能落地的智能化场景最好开发一套系统，能提供固化重复的高校相关制度的问答。这样的智能问答系统带来优点也是显而易见的：

    • 集成在高校的微信公众号服务，学生随时可访问
    • 基本不收集学生隐私，不需要学生主动填写各类测试
    • 对于各类政策主动问答，减少辅导员工作量

    面向学生的智能问答系统，主要解决以下问题场景：学生基本问题，比如校园卡如何缴费，如何申请生源地贷款，如何转专业；考试安排，德育分如何增加，保研政策，考研政策，补考安排，实习政策以及各种证明办理。除此之外，例如学校各办公地点，周边交通等等都可以涵盖。

    这套系统如果完善，还可以推广扩展，比如学校招生季，可以面向家长开发。家长可查询问答最新的各学院招生政策，往届录取政策等等。

    当然开发过程中，需要注意很多地方，比如不能让AI发散，有就有，没有就告知需询问老师。相似知识召回问题，比如学生提问的数学学院，数院，统计学院等等，都应该指向数学与统计学院（只是举例，不同学校可能数院和统计学院是分开的）。

    以下就是该系统开发过程的一些技术和解决方法初探。

    功能特性

    • 📚 知识库构建：支持从本地 PDF/Word 文档自动构建向量知识库
    • 🔄 文档刷新：应用启动时自动扫描，支持手动刷新
    • 💬 多轮对话：支持历史记录和上下文对话
    • 🔒 匹配判断：知识库无匹配时引导联系学校老师，不会自由发挥
    • 🔌 多端接入：同时提供网页、微信小程序和 API，便于嵌入其他 APP
    • ⚙️ 灵活配置：支持大模型 API 和本地部署模型
    • 📝 口语化转化：大模型对口语化提问进行语义理解提炼
    • 🎯 精确匹配：对于语义相近但实质不同的问题（如不同学院的转专业时间），严格区分

    项目优势

    优势 说明
    开箱即用 只需配置模型 API Key，放入文档即可运行，无需复杂训练
    低成本部署 使用 Chroma 轻量级向量数据库，无需额外数据库服务
    多端支持 一套后端同时支持 Web、微信小程序、React Native 等多种前端（微信小程序未测试）
    智能兜底 无匹配结果时自动提示联系相关部门，避免误导用户
    隐私安全 文档在本地处理，数据不上传第三方（除大模型 API 调用）
    易扩展 模块化架构，可方便接入其他模型或添加新功能

    技术栈

    后端

    • FastAPI - Web 框架
    • LangChain - RAG 框架
    • Chroma - 向量数据库
    • PyPDF2 / python-docx - 文档解析

    前端

    • React + TypeScript
    • Ant Design - UI 组件库
    • Zustand - 状态管理
    • Axios - HTTP 客户端

    遇到的问题以及解决方式

    • 用户口语化提问（如”我想转入数学学院”）与知识库中的正式表述（如”数学学院转专业”）语义相近，但向量相似度不够高，导致返回”不清楚”。当前系统直接对原始查询进行向量搜索，缺乏查询理解和提炼层。实际解决时候可采集方案1，如果学院场景复杂，可加入方案2辅助
    方案 优点 缺点 适用场景
    LLM查询重写 智能、通用性强 增加一次LLM调用，延迟+成本 查询类型多样、预算充足
    规则提取 快速、无额外成本 维护映射表，扩展性差 领域固定、查询模式有限
    混合策略 平衡性能和效果 实现稍复杂 生产环境推荐
    多查询扩展 召回率高 搜索次数增加 对召回率要求极高

    • 场景：用户提问”数学学院转专业时间”，但系统回答时参考了管理学院的时间计划。

      根本原因：

      1. 向量搜索基于语义相似度，”数学学院转专业时间”和”管理学院转专业时间”在向量空间上很接近
      2. 召回的文档同时包含数学学院和管理学院的内容
      3. 提示词中没有明确要求LLM只基于特定学院的信息回答
      4. 缺乏学院名称过滤机制

        • 只保留包含该学院名称的文档
        • 如果过滤后无结果，回退到原始结果（避免过度过滤）

    结语

    虽然开篇说了很多面向高校学生的智能化场景落地很少，本次技术研究也没有实际落地，只是纸上谈兵。AI给各行各业带来巨大便利，当AI进入校园时候，希望是给学生和一线老师带来方便为基础，而不是让他们成为某些面子项目的牺牲品，徒增无效的工作量和巨大的负面情绪。
